# 简历部分

  1. 简历中的项目需要有难度的递进，能直观看到能力的提升。写代码考虑到非功能性的需求(比如性能等)。有长远的规划。

# 面试部分

  1. 在面试过程中，描述项目经历时，是在为了某个业务需求推销解决方案，而不是推销自己的编程技能。
  2. 准备一些问题（详见预设问题）
  3. 准备自我介绍

简单的自我介绍（2min） 能够突出：基本信息、工作经历、我突出的技能是什么、我的优势、我有哪些成就和贡献。

面试后需要问面试官 今天的面试怎么样？有什么建议？及时获取反馈。

# 薪酬部分

  1. 讨论薪资的唯一时机是在你们原则上达成一致之后（这时他们已经在你身上投入了大量的时间和金钱，特别是在面试之后）
  2. 聊工资和福利, 这时候可以提价：他们不会因为对你来说很重要，但对于他们相对较小的问题而破坏交易。接受他们的提议，还价，再加上几千，然后坚持下去。
  3. “你以前的薪水是多少?”的意思是“请给我一个理由少给你钱”。不要说具体的薪资期望； 千万不能透露自己的薪资底线
  4. 永远要有还价。在你关心金钱以外的事情时，不要吝啬对薪水的还价。如果他们不能给出更高的薪水，那就谈谈休假吧。

# 介绍一下你做得最出色的项目？

考察候选人在工作中所承担的角色，以及平时工作中是否注意积累和总结

项目最好有自己的思考，有具体的数据支撑

  1. 开头，“有段经历，正好遇到了这个问题……”。（讲清问题背景）
  2. 发展，“一般的做法是做不到的，因为……”。（讲问题的复杂性和挑战）
  3. 高潮，“我一开始用到……后来……”。（讲解决方案的形成过程）
  4. 结局，“成果是……”。（讲结果、影响、意义）

# 离职原因

2015.6-2019.4 在旅游公司做什么工作？什么原因让你想转行做前端的？

刚毕业的两年在公司里做UI和视频剪辑相关的工作，在切图的过程中了解了一些网页制作，然后觉得我也可以，就自学了一段时间累积基础，然后再报培训班进修。 我觉得人不应该满足于现状，而是应该积极进取，如果不能适时挑战自己，就无法最大程度的挖掘自己潜力。

各家公司离职原因？ 第一家单位离职：UI设计和H5页面，接触到html和css,后面为了实现更好的动画效果，自学了js。就这样就慢慢成为了前端，离职原因是想继续朝前端发展。 第二家单位离职：非常感谢会搜给我机会能实践自己的技术，但是确实技术栈单一。业务也不稳定， 第三家单位离职：在这里我开始从零到一负责项目的前端部分。但还是希望能扩展自己的知识面，知道服务于大型互联网企业的代码是怎么出来的。

# 观点

  • 企业主之所以不喜欢程序员，通常是因为难以在企业和程序员之间建立价值联系。
  • 可以创造创造业务价值，而不是为了编写程序的工程师，能够为企业增加收入或者降低成本。
  • 客观看待加班问题？资本富集的地方，每个人都得加班，人作为劳动力，也是资本的一种。过去汽车制造业的机械工程师也是经常加班。
  • 技术不是源头，需求才是。一切技术问题，都要服从产品交付和市场反馈
  • 为了追求商业目标，可以做出技术上的牺牲（比如在接到需要尽快落地的需求时，在性能与开发效率之间选择了开发效率。）
  • 不迷信技术，IT是工科，不是理科。做得好不好不取决于智商，而取决于是否愿意不断读书不断学习不断积累。(原因：能大规模商用的技术，都不需要智商，否则这种技术就不可能规模化。)
  • 多练习沟通，训练自己能够快速、简明、自信地向一个陌生人解释你如何创造价值。删掉总是出现的一些技术术语。在任何适当的抽象层次上解释自己所做的事情。
  • 如果你也参与了和团队一起努力，而且团队努力取得了成功，正确的说法不是“我的一切都归功于我的团队”，除非你的立场是，所有人都能理解你为了谦虚而这样讲。你应该这样说：“我很荣幸能够帮助我的团队完成 $YOUR_SPECIALTY 方面的工作。对着镜子说一千遍，直到你能板着脸说出来。
  • 认为让自己变得不可替代的手段是通过写文档。描述一下自己是怎么写文档的：
    • 明确需求和测试用例(产品和测试的意图)，在需求分析与测试用例宣讲之后。
    • 画时序图/数据流程图：思考设计的功能模块之间的关系。
    • 有(可能)频繁变化的模块/需求，则使用设计模式(画 UML 类图)；没有频繁变化的模块/需求，一定不要用设计模式。
    • 在一个功能模块中，逻辑比较复杂的地方，画流程图
    • 模块和模块之间有没有需要明确的协议？如果有，把协议写出来。
